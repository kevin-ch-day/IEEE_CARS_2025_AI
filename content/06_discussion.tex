\section{Discussion and Limitations}

The cutoff-window model addresses a practical research problem: exact live installed-build status is not stable enough to be the study gate when consumer apps update during collection. Without a recent evidence window, the study can enter a loop where evidence is valid when collected but appears operationally stale after an app-store update. This study avoids that loop by treating recent selected-build evidence as valid when it falls within the final 14-day cutoff window and retains package/version provenance.

This approach does not weaken the evidence model. It makes the evidence model more explicit. Every app-level row is tied to version identity, static run identifiers, dynamic run identifiers, APK hashes, packet-capture availability, collection timestamps, and quality-assurance provenance. Readers can distinguish the selected evidence window and runtime class coverage for each app instead of relying on a fragile same-day installed-build label.

App update churn is therefore not evidence loss. If a selected evidence bundle was collected from version code X inside the cutoff window and the app later updated to version code Y, the version-X evidence remains valid for claims about the selected version-X bundle. Version Y becomes useful for future refresh or longitudinal comparison, but it does not invalidate the recent version-X measurements.

Static and dynamic evidence complement each other in this model. Static analysis identifies declared capabilities, component exposure, network/security posture, strings, endpoint hints, and risk-relevant app structure. Dynamic analysis shows which behaviors and services appear during baseline, QFG, and interactive capture windows. Linking the two through static run identifiers, dynamic run identifiers, APK hashes, and package/version identity lets the paper discuss observed behavior without losing sight of the app's declared and embedded surface.

The QFG category is especially important for modern social and media apps. A no-touch foreground app can still load feeds, refresh content, synchronize services, or maintain communication state. Treating those captures as strict idle would overstate baseline quietness. Treating them as invalid would discard meaningful app-driven behavior. This paper instead reports QFG as its own class: valid no-touch foreground evidence, but not strict idle.

The study has several limitations. It is bounded by its 15-app cohort, single device, Android 15 environment, selected user scenarios, and July 2026 cutoff window. The final cutoff also does not mean that app-store updates stopped or that every possible runtime behavior was exercised. The correct claim is that all 15 apps had recent build-backed evidence at cutoff.

Interactive-depth coverage also varies across apps. Some selected evidence bundles contain stronger interactive coverage, while others are supported by baseline, QFG, or limited interactive evidence. The paper therefore avoids claims that require identical interactive depth across every app.

Finally, the study is bounded by a single-device, Android 15, non-root setting. That setting is deliberate and realistic for non-invasive behavioral measurement, but it limits claims about other Android versions, OEMs, devices, rooted instrumentation, and alternate account states. Static detector caveats, resource parsing caveats, unresolved service mappings, and dynamic signal limitations should be retained in future expanded analyses.
