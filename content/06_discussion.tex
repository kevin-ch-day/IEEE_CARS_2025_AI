\section{Discussion}

\subsection{Evidence Stability Under App-Update Churn}

The 14-day evidence window separates research readiness from the continuously changing operational queue. When a selected app build is captured within the window and linked to valid static and runtime evidence, a subsequent app-store update does not invalidate those observations. Instead, the newer build becomes a candidate for a future comparison wave. This approach prevents recent, version-backed measurements from being discarded solely because the installed application changed after collection, while preserving the package, version, APK, static-run, dynamic-run, and quality-assurance provenance needed to interpret each result.

\subsection{Interpreting Static and Runtime Evidence}

Static and runtime evidence describe related but different aspects of Android app posture. Static analysis characterizes declared capabilities, packaged attack surface, exported components, permissions, configuration choices, and detector findings. Runtime analysis characterizes the network activity observed during selected idle, quiescent foreground, and interactive execution scenarios. A high static finding count does not necessarily imply broad runtime activity, while limited observed runtime activity does not demonstrate low privacy or security exposure. The integrated results should therefore be interpreted as cross-layer characterization rather than evidence of causation, prediction, exploitability, or malicious behavior.

\subsection{App-Level Profiles and Review Prioritization}

The app-level profiles illustrate why both evidence layers are needed. Static-heavy apps such as Snapchat and WhatsApp expose substantial packaged surfaces even when the selected runtime scenarios do not produce the broadest domain diversity. Lower-static but higher-runtime cases such as CNN and BBC News demonstrate that interactive network activity can broaden even when static finding volume is lower than that of large social platforms. Apps such as Facebook, Instagram, TikTok, and X exhibit higher values across both layers and may therefore warrant deeper manual examination. These profiles are intended to support review prioritization, not to function as composite risk scores or formal security rankings.

\subsection{QFG Separation and Practical Implications}

Separating quiescent foreground (QFG) captures from strict idle captures is an important methodological distinction. Social, news, and messaging apps may continue loading content, synchronizing services, or maintaining communication state while foregrounded but untouched. Counting this activity as strict idle would overstate baseline quietness, whereas discarding it would remove meaningful app-driven behavior. Reporting QFG as a separate evidence class preserves both interpretations. More broadly, the findings indicate that Android app assessment should not be reduced to either static inspection or runtime traffic observation alone. Build-aligned static and runtime evidence allows practitioners to compare declared exposure with observed behavior without combining measurements from different application versions.
