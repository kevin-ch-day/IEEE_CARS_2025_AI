\section{Discussion}

\subsection{Evidence Stability Under App-Update Churn}

The 14-day evidence window separates research readiness from the continuously changing operational queue. When a selected app build is captured within the window and linked to valid static and runtime evidence, a later app-store update does not invalidate those observations; the newer build instead becomes a candidate for future longitudinal comparison. The window preserves recency while keeping each claim tied to concrete package version, APK hash, static run, dynamic run, PCAP-derived features, and QA provenance.

\subsection{Interpreting Static and Runtime Evidence}

Static and runtime evidence describe related but different dimensions of Android app posture. Static analysis characterizes declared capabilities, packaged attack surface, exported components, permissions, configuration choices, and detector findings. Runtime analysis characterizes network activity observed during selected strict-idle, QFG, and interactive scenarios. A high static finding count does not necessarily imply broad runtime activity, while limited observed runtime activity does not demonstrate low privacy or security exposure. The integrated results should therefore be interpreted as cross-layer characterization rather than evidence of causation, prediction, exploitability, or malicious behavior.

\subsection{App-Level Profiles and Review Prioritization}

The app-level profiles illustrate concrete triage implications. Static-heavy apps such as Snapchat and WhatsApp expose substantial packaged surfaces even when selected runtime scenarios do not produce the broadest domain diversity. Lower-static but higher-runtime cases such as CNN and BBC News suggest follow-up scenario and service-family review, while Facebook, Instagram, TikTok, and X exhibit higher values across both layers and may warrant deeper manual examination. These profiles are not composite scores or formal security rankings; they preserve disagreement between layers as a signal for defensible follow-up.

\subsection{QFG Separation and Practical Implications}

Separating QFG captures from strict-idle captures is an important methodological distinction. Social, news, and messaging apps may continue loading content, synchronizing services, or maintaining communication state while foregrounded but untouched. Counting this activity as strict-idle would overstate baseline quietness, whereas discarding it would remove meaningful app-driven behavior. Keeping QFG separate allows the paper to describe app-driven foreground behavior without claiming that it represents user inactivity in the strictest sense.
