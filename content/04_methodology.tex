\section{Methodology}

\subsection{Cohort and Device Setting}

The study used the Research Dataset Beta cohort, a 15-app set analyzed on a non-root Motorola moto g 5G 2024 device running Android 15. The cohort includes social, media, messaging, and news applications selected to expose common mobile behavioral surfaces: feeds, messaging, public content loading, calls, media discovery, account-linked services, and app-driven foreground network activity. The non-root physical-device setting is part of the study design because it reflects an analyst-observable environment without privileged instrumentation, application modification, TLS interception, or root-only filesystem access.

\subsection{Static Evidence}

Static evidence was generated from harvested APK artifacts and preserved as part of each app's build-backed evidence bundle. The harvested inputs include base APKs and split APK members associated with the selected package version. Static outputs include package and version identity, APK hashes, static run identifiers, manifest and permission surfaces, exported component posture, network and storage flags, string/resource findings, and static-to-dynamic handoff context. These artifacts provide the vocabulary used to interpret runtime observations, including privacy-sensitive capabilities, embedded endpoint indicators, SDK signals, and component exposure.

\subsection{Runtime Evidence}

Dynamic evidence was generated through foreground capture scenarios and package-filtered network observation. Dynamic runs preserve dynamic run identifiers, run profile, package/build identity, packet-capture availability, quality-assurance verdict, scenario class, and static-plan linkage. Runs were classified into strict idle baselines, quiescent foreground baselines, and interactive captures. These classes answer different questions: strict idle measures quiet foreground behavior, quiescent foreground captures app-driven activity without direct interaction, and interactive captures exercise user-facing features or communication flows.

\subsection{Cutoff-Window Evidence Model}

Dynamic collection was closed at a fixed cutoff because consumer Android applications updated during the study faster than a full same-day installed-build collection wave could be completed. Evidence was therefore assembled as app-level, build/version-backed bundles from the final 14-day cutoff window, preserving package name, version code/name, static run identifiers, dynamic run identifiers, APK hashes, packet-capture availability, collection timestamps, and quality-assurance provenance. App-store updates during or after that window were treated as normal churn, not evidence loss. Claims in this paper use the selected recent build-backed evidence bundle for each app and do not require every app to remain on the same installed build after collection.

Strict idle baselines, quiescent foreground baselines, and interactive captures were reported as separate evidence classes. Quiescent foreground captures represent valid no-touch foreground observations in which the app continued app-driven network activity; they were not counted as strict idle. This distinction is important for apps whose foreground surfaces continue loading feed, media, messaging, or service state even when the operator does not touch the screen.

\subsection{Privacy Boundary}

The packet-capture privacy model is limited to behavioral and transport-level evidence. The study uses package-filtered PCAP metadata and derived indicators; it does not export message bodies, content payloads, cookies, authorization headers, private account content, or decrypted TLS contents. This boundary is consistent with the non-root, no-MITM, no-Frida, no-TLS-decryption design and supports analysis of traffic shape, endpoint context, service activation, and capture health without collecting application-layer private content.
