\section{Background}

Android app security evidence spans manifest declarations, requested permissions, exported components, packaged resources, embedded libraries, native code, storage configuration, and network-security settings. Permissions protect restricted data and actions, while platform guidance encourages minimizing data access and protecting inter-application communication \cite{androidPermissions2026,androidPrivacyChecklist2026,androidSecurityBestPractices2026}. These static surfaces are useful indicators, but they do not prove runtime behavior.

Static analysis is repeatable and versionable. It can inspect packaged APK evidence before execution and can be linked to exact package/version identity. Prior work on manifest correctness and Android SAST tooling shows both the value and limits of static inspection: structural exposure can be measured, but tool coverage, detector semantics, and false-positive boundaries must be reported carefully \cite{yang2022maniscope,zhu2024sastandroid}. In this paper, static findings are treated as exposure indicators, not as proof of exploitability.

Runtime analysis complements static evidence by observing behavior under controlled conditions. For Android network studies, packet captures can support traffic-shape, service-family, DNS, TLS-fingerprint, and protocol-prevalence measurements. They do not, in a non-root and no-MITM setting, reveal encrypted payload semantics or private message contents. We therefore report runtime metadata and derived indicators only.

Modern Android distribution also affects reproducibility. Google Play commonly delivers device-specific APK sets from Android App Bundles, including base and split APKs \cite{androidAppBundles2026}. A reproducible study must retain package name, version code/name, APK hashes, split membership, static run identifiers, dynamic run identifiers, capture timestamps, and quality-assurance state. This build-backed linkage is the foundation for the static-runtime alignment used below.
