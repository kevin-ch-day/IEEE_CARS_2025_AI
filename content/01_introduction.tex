\section{Introduction}

Android consumer applications are routine gateways for news, social networking, messaging, professional communication, and personalized content. These apps are not necessarily malicious, but they combine sensitive permissions, exported components, embedded SDKs, cloud services, advertising systems, and encrypted network traffic. Android's permission model and Google Play's Data safety section provide user-facing and platform-facing controls \cite{androidPermissions2026,googlePlayDataSafety2026}; however, technical claims about privacy and security still require evidence tied to a specific app build.

This paper asks: \emph{How do exact-build-aligned static exposure indicators and runtime network observations jointly characterize privacy and security posture across Android consumer applications?} Three subquestions guide the analysis: RQ1, what static exposure patterns appear across selected app builds; RQ2, what runtime behavior and evidence-class patterns appear across the same builds; and RQ3, what relationships or divergences emerge when static and runtime evidence are aligned at the build level?

Prior Android studies show why a single evidence stream is insufficient. Static analysis can identify manifest exposure, permission posture, exported components, packaged resources, SDK indicators, and network-security settings, but it cannot establish how an app behaves when executed. Dynamic analysis can observe packet timing, traffic volume, endpoint families, and interaction-driven changes, but runtime observations are scenario-bounded and depend on app version, device state, network conditions, and user stimulation \cite{sutter2024dynamic}. Recent privacy-disclosure work further shows that developer labels, SDK configurations, and observed behavior can diverge \cite{arkalakis2024datasafety,khandelwal2024privacylabels,zhang2024privacysdks}.

We present a 15-app empirical study of mainstream Android consumer apps using ScytaleDroid, a database-backed analysis platform that links APK harvests, base and split APK identity, APK hashes, static findings, runtime evidence packs, PCAP availability, and quality-assurance state. The study includes social/content, messaging, news, and professional-social apps. The empirical unit is an app-level, selected-build evidence bundle rather than a mutable installed-app label.

The main methodological challenge is app-update churn. Popular consumer apps can update faster than a full same-day measurement wave can finish. This study therefore uses a fixed 14-day recent-evidence window as a provenance control. The window does not replace the research question; it defines which version-backed observations are eligible for analysis. At cutoff, all 15 apps had build-backed static and runtime evidence with no true evidence holes.

The contributions are:
\begin{itemize}
    \item a 15-app static/runtime Android consumer-app analysis with exact selected-build provenance;
    \item a recent-evidence cutoff model that prevents routine app-store updates from invalidating versioned measurements;
    \item separate reporting of strict idle, quiescent foreground (QFG), and interactive runtime classes;
    \item integrated descriptive profiles that compare static exposure volume with runtime behavior without creating an opaque composite risk score.
\end{itemize}

The remainder of this paper is organized as follows. Section~II provides background. Section~III reviews related work and states the gap. Section~IV describes the methodology. Section~V presents results. Section~VI discusses limitations. Section~VII concludes.
